# ALZ Glossary

Accumulating glossary for the ALZ study notes. Seeded from `references/glossary-seed.md`, then
extended with terms encountered while reading repositories. English, concise, one term per row.

| Term | Definition |
|------|------------|
| Azure Landing Zone (ALZ) | A pre-configured, scalable, and secure Azure environment that follows Cloud Adoption Framework (CAF) best practices, providing the foundation (identity, governance, networking, security) for hosting workloads. |
| Cloud Adoption Framework (CAF) | Microsoft's guidance and best practices for adopting Azure; ALZ is the technical implementation of its "Ready" landing zone guidance. |
| Enterprise-Scale | The CAF architecture pattern (and reference implementation) for large-scale, governed Azure environments; the conceptual basis for ALZ. |
| Management Group (MG) | A container above subscriptions used to organize them into a hierarchy for applying governance (policy, RBAC) at scale. |
| Tenant Root Group | The top-most management group in an Entra ID (Azure AD) tenant; all other MGs and subscriptions sit beneath it. |
| Intermediate Root Management Group | The custom top-level MG created for ALZ (e.g. named after the org), under which Platform / Landing Zones / etc. live. |
| Subscription | The billing and access-control boundary in Azure that contains resource groups and resources. |
| Subscription Vending | The automated process of creating, configuring, and placing new subscriptions (networking, RBAC, budget) into the right management group — standardized by `avm/ptn/lz/sub-vending`. |
| Platform (landing zone) | Management groups/subscriptions hosting shared services: Management, Connectivity, Identity. |
| Management subscription | Platform subscription hosting monitoring/management services (Log Analytics, Automation, Sentinel). |
| Connectivity subscription | Platform subscription hosting shared networking (hub VNet, firewall, gateways, DNS). |
| Identity subscription | Platform subscription hosting identity services (e.g. domain controllers) and related network. |
| Landing Zone (application) | Subscriptions where workloads are deployed, typically under "Corp" or "Online" management groups. |
| Corp (management group) | Landing zones for workloads that require connectivity to on-premises / corporate network (no direct internet exposure). |
| Online (management group) | Landing zones for internet-facing workloads that may not need corporate connectivity. |
| Hub-and-Spoke | A network topology with a central hub VNet (shared services: firewall, gateways) peered to multiple spoke VNets (workloads). |
| Virtual WAN (vWAN) | A managed, Microsoft-operated networking hub alternative to self-managed hub-and-spoke, providing global transit connectivity. |
| Azure Firewall | A managed, stateful network security service typically deployed in the hub to inspect and control traffic. |
| VNet Peering | A connection linking two virtual networks so resources can communicate as if on the same network. |
| Policy (Azure Policy) | A governance service that enforces rules (audit/deny/deployIfNotExists) on resources at a given scope. |
| Policy Definition | The rule itself (what to evaluate and what effect to apply). |
| Policy Initiative (Set) | A grouping of related policy definitions assigned together as one unit. |
| Policy Assignment | Binding a policy/initiative to a scope (MG/sub/RG) so it takes effect. |
| Archetype | (caf-enterprise-scale concept) A named bundle of policy assignments, role assignments, and policy/role definitions applied to a management group to express its governance posture. |
| DeployIfNotExists (DINE) | A policy effect that auto-deploys a remediation resource when a condition isn't met (e.g. enable diagnostics). |
| RBAC (Role-Based Access Control) | Authorization system assigning roles (permission sets) to identities at a scope. |
| Role Definition | A named collection of permissions (built-in or custom). |
| Role Assignment | Binding a role definition to a principal at a scope. |
| Managed Identity | An Entra ID identity managed by Azure for services to authenticate without stored credentials. |
| Log Analytics Workspace | The data store/query engine for Azure Monitor logs; central to ALZ management subscription. |
| Azure Verified Module (AVM) | Microsoft's standardized, supported, well-tested IaC module library (Bicep & Terraform), categorized as Resource (res), Pattern (ptn), and Utility (utl) modules. |
| AVM Resource Module (res) | An AVM module that deploys a single Azure resource type with sensible secure defaults. |
| AVM Pattern Module (ptn) | An AVM module that orchestrates multiple resources for a scenario (e.g. `lz/sub-vending`). |
| Bicep | Azure's domain-specific declarative IaC language that transpiles to ARM JSON. |
| ARM template | Azure Resource Manager JSON template describing resources to deploy declaratively. |
| Terraform (AzureRM) | HashiCorp's IaC tool with the AzureRM provider used to manage Azure resources. |
| Module (IaC) | A reusable, parameterized unit of infrastructure (a `.bicep` file, a Terraform `module`, or a nested ARM deployment). |
| Bootstrap | The initial setup phase (state storage, CI/CD identity via OIDC, pipelines) that precedes deploying the platform. |
| Deployment Scope | The level at which a template deploys: management group, subscription, or resource group — significant in ALZ. |
| Diagnostic Settings | Configuration that routes resource logs/metrics to a destination (e.g. Log Analytics); often enforced via DINE policy. |
| ALZ Accelerator | The end-to-end "getting started" experience that bootstraps a customer's VCS + Azure for ALZ and wires in a starter module; driven by the `ALZ` PowerShell module. |
| ALZ PowerShell Module | The `ALZ` PowerShell module (this repo, F3). Exposes cmdlets such as `Deploy-Accelerator` that orchestrate downloading modules, gathering inputs, rendering config, and running Terraform. |
| Bootstrap module | A downloaded Terraform module (default repo `Azure/accelerator-bootstrap-modules`) that provisions CI/CD plumbing: VCS repos, identities/OIDC, Terraform state storage, and pipelines. Even Bicep deployments use Terraform here. |
| Application LZ CI/CD bootstrap (B7/B8) | The intended AVM pattern pair `avm-ptn-alz-application-landing-zone-cicd-bootstrap-{azure-devops,github}` — the **per-application** analogue of the platform bootstrap module, standing up an app team's repo/pipeline/identity (ADO for B7, GitHub for B8). As of 2026-06-17 both are **scaffold-only AVM templates** (identical, dummy `azurerm_resource_group.TODO`), not yet implemented. |
| AVM template scaffold | A repo generated from the AVM Terraform module template but not yet implemented: `main.tf` carries the placeholder `azurerm_resource_group.TODO`, README is the generic template, inputs are the standard AVM interfaces (`location`/`name`/`resource_group_name` + lock/role_assignments/private_endpoints/…). A signal the module is pending build-out (cf. B7/B8). |
| Starter module | The actual ALZ platform IaC (Terraform or Bicep) that the accelerator commits into the customer's repo for the platform deployment. Selected via `starter_module_name`. |
| Bootstrap type / module name | A named bootstrap variant (e.g. GitHub vs Azure DevOps) selected via `bootstrap_module_name`; each maps to a set of compatible starter modules in the bootstrap config. |
| Input config (PowerShell) | A normalized `PSCustomObject` where every property is `{ Value, Source, Sensitive }`. Built by merging config files (yaml/json/tfvars), env vars (`ALZ_*`/`TF_VAR_*`), and cmdlet parameters. |
| Config Source | The provenance tag on an input value: `yaml` / `json` / `.tfvars` (from a file), `parameter` (cmdlet arg), `calculated` (computed at runtime), or env-sourced. Used to drive precedence and file emission. |
| iac_type | The required input selecting the IaC flavor: `terraform`, `bicep`, or `bicep-classic`. Determines how starter parameters and output files are rendered. |
| HCL Parser Tool | A downloaded helper binary (e.g. `v0.6.0`) used to convert Terraform `.tf`/`.tfvars` (HCL) into JSON so PowerShell can read variable declarations and tfvars inputs. |
| terraform.tfvars.json | The JSON-form Terraform variables file the module writes for the bootstrap (and, for Terraform starters, the starter) so Terraform consumes the rendered inputs non-interactively. |
| Module version data | A `.alz-version-data.json` file in the output folder tracking the downloaded bootstrap/starter release tags, enabling reuse and `-upgrade` state-file migration. |
| Override folder path | A local source folder (`*_module_override_folder_path`) used instead of a GitHub download; the release tag becomes `local`. Useful for air-gapped/dev scenarios. |
| ShouldProcess | PowerShell's `SupportsShouldProcess` mechanism enabling `-WhatIf`/`-Confirm`; the module gates side-effecting work behind `$PSCmdlet.ShouldProcess(...)`. |
| SubscriptionCreator role | A built-in Azure **billing** role (ID `a0bcee42-bf30-4d1b-926a-48d21664ef71`) that lets a principal create subscriptions under a billing scope; a prerequisite for subscription vending. |
| Enterprise Agreement (EA) | A Microsoft volume licensing/billing agreement; its billing scope is `billingAccounts/<acct>/enrollmentAccounts/<enroll>`. |
| Microsoft Customer Agreement (MCA) | A modern Microsoft billing agreement; its billing scope is `billingAccounts/<acct>/billingProfiles/<profile>/invoiceSections/<section>`. |
| Plan mode (`-PlanMode`) | A dry-run switch on the `Remove-*` cleanup cmdlets that prints what would be deleted without making changes. |
| Scenario (accelerator) | A predefined starter input set for the Terraform accelerator, selected by number and mapped via `TerraformScenarios.json` to a tfvars file. |
| Runner group / Agent pool | CI/CD compute groupings (GitHub Actions runner groups / Azure DevOps agent pools) created by the bootstrap and removable via the `Remove-*Accelerator` cmdlets. |
| Bootstrap root module | A Terraform root module in `accelerator-bootstrap-modules/alz/*` (`azuredevops`, `github`, `local`) that is the actual entry point Terraform applies for one bootstrap target. |
| ALZ-Powershell.config.json | The bootstrap config in `accelerator-bootstrap-modules/.config/`; maps `bootstrap_module_name` → root-module `location` and `iac_type` → starter repo/artifact. The contract between the PowerShell driver (F3) and the Terraform bootstrap (F2). |
| Federated credential (OIDC) | An Entra ID federated identity credential bound to a user-assigned managed identity, letting a VCS pipeline (GitHub/ADO) authenticate to Azure without storing secrets. The bootstrap's auth mechanism. |
| Terraform state backend | The Azure storage account + blob container the bootstrap creates (only when `iac_type == terraform`) to hold remote Terraform state for the platform deployment. |
| Self-hosted agent/runner | Optional container-based CI/CD compute (built as an image in an ACR and run as a container instance, usually on a private VNet) provisioned by the bootstrap's `azure` module. |
| Template repository | An optional separate VCS repository the bootstrap can create to hold reusable CI/CD pipeline templates, kept apart from the platform config repository. |
| Local bootstrap target | The `alz/local` root module: renders the starter files to the local file system (+ `deploy-local.ps1`) and optionally creates the Azure backend — for restricted/dev scenarios without a VCS. |
| Platform Landing Zone (starter) | The `platform_landing_zone` Terraform starter module (F1) that deploys the ALZ platform by orchestrating the AVM pattern modules (management groups/policy, management resources, connectivity). |
| config-templating module | The central `module.config` in the Terraform starter that transforms raw tfvars (+ `lib/` overrides) into per-module `settings` objects consumed by every child module. |
| Provider alias (multi-subscription) | Terraform `provider "azurerm"` aliases (e.g. `azurerm.management`, `azurerm.connectivity`) used to deploy different parts of the platform into different platform subscriptions in one run. |
| ALZ Library customization (`lib/`) | Local overrides layered on the upstream `Azure-Landing-Zones-Library` (G1) — archetype/policy/architecture customizations (e.g. sovereign `sovereign_l1/l2/l3`) consumed by `avm-ptn-alz`. |
| Network Virtual Appliance (NVA) | A third-party/virtual network appliance (firewall/router) deployed in the hub instead of Azure Firewall; offered as `*-nva` starter scenarios. |
| `moved` block | A Terraform construct that remaps a resource/module to a new state address without destroy/recreate; used widely in the starter to preserve state across module-version refactors. |
| Starter scenario | A predefined tfvars profile under the starter's `examples/` folder (e.g. single/multi-region, NVA, management-only, SMB) selected during accelerator setup. |
| Architecture definition | A `*.alz_architecture_definition.{json,yaml,yml}` file (in the ALZ Library / `lib/`) describing the management-group tree: each MG's `id`, `parent_id`, `display_name`, `exists`, and assigned `archetypes`. Consumed by the `alz` provider. |
| Archetype (ALZ Library) | A named bundle of policy assignments, policy/set definitions, and role assignments/definitions attached to a management group via the architecture (e.g. `root`, `platform`, `management`, `connectivity`, `corp`, `online`, `sandbox`, `identity`). Determines that MG's governance posture. |
| `alz` provider | The Terraform `azure/alz` provider (G3). Its `alz_architecture` data source resolves an architecture + archetypes from the ALZ Library into concrete policy/role/MG objects at plan time. Backed by `alzlib` (G2). |
| AzAPI provider | The `Azure/azapi` Terraform provider that talks directly to ARM REST APIs (with retry logic). `avm-ptn-alz` uses it instead of `azurerm` for speed/reliability and to create MG-scoped policy/role resources. |
| `_dependencies` variables | Inputs (`management_groups_dependencies`, `policy_assignments_dependencies`, `policy_role_assignments_dependencies`) used to order `avm-ptn-alz` against external resources, since the `alz` provider data source forbids `depends_on`/unknown values. |
| assignPermissions (metadata) | A policy parameter metadata flag telling the tooling to create the role assignments a DINE/Modify policy's managed identity needs; `avm-ptn-alz` can set/unset it to fix mis-authored policies. |
| Subscription placement | Associating a subscription with a management group; `avm-ptn-alz` does this via `azapi_resource_action` with a configurable destroy behavior (`parent`/`intermediate_root`/`custom`/`default`). |
| .alzlib | The local cache folder into which the `alz` provider downloads the referenced ALZ Library; must be git-ignored. |
| Data Collection Rule (DCR) | An Azure Monitor resource (`Microsoft.Insights/dataCollectionRules`) defining what telemetry to collect and where to send it; `avm-ptn-alz-management` ships three: change tracking, VM insights, Defender for SQL. |
| Azure Monitor Agent (AMA) | The unified Azure Monitor data-collection agent; ALZ deploys a user-assigned identity (`uami-ama`) for it and DCRs that it uses. Replaces the legacy Log Analytics agent (MMA). |
| Change Tracking & Inventory | An Azure Monitor solution that tracks changes to files, registry, software, and services on machines; configured via a dedicated DCR in the management module. |
| VM Insights | An Azure Monitor solution collecting VM performance counters + the Dependency Agent (ServiceMap) for the map experience; configured via a dedicated DCR. |
| Microsoft Sentinel | Azure's cloud-native SIEM/SOAR; onboarded onto the Log Analytics workspace via `Microsoft.SecurityInsights/onboardingStates` (off by default in the management module). |
| Automation Account | An Azure Automation resource for runbooks/DSC/update management; optionally created and linked to the workspace (`linked_automation_account_creation_enabled`, default off). |
| Log Analytics Solution Plan | A management/insights solution (e.g. `OMSGallery/ContainerInsights`, `VMInsights`) installed onto a Log Analytics workspace. |
| Brownfield / BYO (bring-your-own) | A deployment mode reusing pre-existing resources (e.g. an existing Log Analytics workspace or resource group) instead of creating new ones, toggled by `*_creation_enabled` flags. |
| Azure Bastion | A managed jump-host service deployed in the hub (`AzureBastionSubnet`) providing secure RDP/SSH to VMs over TLS without public IPs on the VMs. |
| VPN Gateway | A virtual network gateway (`GatewaySubnet`) for site-to-site / point-to-site IPsec VPN connectivity to on-premises or clients. |
| ExpressRoute Gateway | A virtual network gateway for private, dedicated ExpressRoute circuit connectivity to on-premises (no traffic over the public internet). |
| GatewaySubnet | The reserved, specifically-named subnet (`GatewaySubnet`) that VPN/ExpressRoute gateways must be deployed into. |
| terraform-azurerm-vnet-gateway (B9) | The classic (pre-AVM) Terraform module `Azure/vnet-gateway/azure` that deploys an Azure Virtual Network Gateway (VPN or ExpressRoute) + `GatewaySubnet`, public IPs, optional route table, local network gateways, ER circuit peerings, and gateway connections. **Obsolete** — superseded by `terraform-azurerm-avm-ptn-vnetgateway`. |
| Local Network Gateway (LNG) | An Azure resource (`azurerm_local_network_gateway`) representing an on-premises VPN endpoint (its public IP/FQDN + address spaces + optional BGP), used as the remote end of a site-to-site VPN connection. In B9, entries with a non-null `id` are referenced (existing); `id == null` entries are created. |
| Virtual Network Gateway Connection | An Azure resource (`azurerm_virtual_network_gateway_connection`) linking a VNet gateway to a remote endpoint — a local network gateway (IPsec/S2S), another VNet (Vnet2Vnet), or an ExpressRoute circuit. B9 derives these from maps keyed `lgw-*` / `erc-*`. |
| BridgeCrew Yor (tracing tags) | A tagging tool (`bridgecrewio/yor` + `yorbox`) that injects provenance tags (git commit/file/org/repo + a `yor_trace` UUID) onto resources. B9 merges these `avm_*`-prefixed tags when `tracing_tags_enabled = true`. |
| Private DNS Resolver | An Azure service (inbound/outbound endpoints + forwarding rulesets) enabling DNS resolution between on-premises and Azure private DNS zones without custom DNS VMs. |
| DDoS Protection Plan | A tenant/region-shared Azure resource providing enhanced DDoS mitigation, optionally associated with hub VNets. |
| NAT Gateway | An Azure resource providing scalable outbound SNAT for subnets; can be attached to hub subnets (and coexist with Azure Firewall). |
| Mesh peering | The full-mesh VNet peering between multiple regional hub VNets, with route tables routing each hub to the others' address spaces via the local Azure Firewall. |
| AVM resource module (res) composition | The pattern where an AVM `ptn` module declares almost no resources itself and instead wires together many AVM `res` modules (e.g. virtualnetwork, azurefirewall, bastionhost, dnsresolver) + local sub-modules. |
| Virtual Hub | A Microsoft-managed regional network hub inside a Virtual WAN; the vWAN equivalent of a self-managed hub VNet. Spokes connect to it via virtual hub connections. |
| Secured Virtual Hub | A Virtual Hub with an Azure Firewall (SKU `AZFW_Hub`) deployed *inside* it (via the firewall's `virtual_hub` block); traffic is steered through the firewall using Routing Intent. |
| Routing Intent | The Virtual WAN mechanism that forces hub traffic (`Internet` / `PrivateTraffic` destinations) through a next-hop Azure Firewall, replacing user-defined route tables (UDRs). |
| Virtual Hub Connection | An `azurerm_virtual_hub_connection` linking a spoke VNet to a Virtual Hub (the vWAN equivalent of VNet peering to a hub). |
| Sidecar Virtual Network | A regular VNet connected to a Virtual Hub used to host resources that cannot live in a managed hub (Azure Bastion, Private DNS Resolver, AVM gateway subnets). |
| VPN Site | A Virtual WAN object representing an on-premises/branch location (its links, BGP, device info) that connects to the vWAN S2S VPN gateway. |
| P2S / S2S VPN Gateway (vWAN) | Virtual WAN-native point-to-site / site-to-site VPN gateways deployed in a Virtual Hub for client and branch connectivity. |
| Route Map (vWAN) | A Virtual WAN object (`Microsoft.Network/virtualHubs/routeMaps`) that transforms routes (AS-path / community / prefix) on hub connections. |
| BYO Virtual WAN | Bring-your-own mode where an existing Virtual WAN resource id is supplied so the module attaches hubs/gateways instead of creating a new vWAN. |
| BGP Connection (vHub) | An `azurerm_virtual_hub_bgp_connection` peering a spoke-hosted NVA's BGP router directly with the Virtual Hub router (peer ASN must not be 65515). |
| ALZ Library | The `Azure/Azure-Landing-Zones-Library` repo (G1): the authoritative, versioned **data** source of ALZ policy/role assets, archetypes, architectures, and policy default values. Consumed by `alzlib`/`alz` provider/B1. Not IaC. |
| Platform library | A folder under `platform/` in the ALZ Library — one self-contained, independently versioned library. The repo ships three: `alz`, `amba`, `slz`. |
| Archetype definition (file) | A `*.alz_archetype_definition.{json,yaml,yml}` file: a `name` plus lists of `policy_assignments`, `policy_definitions`, `policy_set_definitions`, `role_definitions` referenced **by their `.name`**. The bundle a management group can apply. |
| Archetype override (file) | A `*.alz_archetype_override.{json,...}` file: `name` + `base_archetype` + `*_to_add` / `*_to_remove` lists. An additive delta letting you customize an archetype without forking it (used by SLZ and F1 `lib/`). |
| Resource definition (asset) | An ALZ Library asset that mirrors a **native Azure resource schema**: policy definition / policy set / policy assignment / role definition. The client rewrites scope-specific values (ids, scopes, locations) at deploy time. |
| Construct (asset) | An ALZ Library file that is library-specific glue (archetype, archetype override, architecture, metadata, policy default values) — has a custom JSON schema in `schemas/`. |
| Policy default values | The `*_policy_default_values.json` file mapping a named **`defaultName` token** (e.g. `log_analytics_workspace_id`) to the policy assignments + parameter names it fills. The contract that injects B2's monitoring resource ids into ALZ policy. |
| `defaultName` token | A single named entry in policy default values (e.g. `ddos_protection_plan_id`); a client supplies one value and the provider fans it out to every assignment/parameter the token lists. |
| Library reference | A `{ path, ref }` entry in the `alz` provider's `library_references` (e.g. `path = "platform/alz", ref = "2026.04.2"`) that pins a specific library at a specific release tag. |
| `library_metadata.json` | A construct file declaring a library's name and its dependencies on other libraries (lets libraries layer/compose). |
| AMBA library | `platform/amba`: the **Azure Monitor Baseline Alerts** library — alert policy definitions/initiatives (e.g. `amba_root`, `amba_connectivity`) stackable on top of `alz`. |
| SLZ library | `platform/slz`: the **Sovereign Landing Zone** library — inherits the `alz` archetypes and adds `sovereign_l1_controls` / `_l2_controls` / `_l3_controls` and `public` for sovereignty/compliance. |
| Sovereign controls archetypes | The `sovereign_l1/l2/l3_controls` archetypes (SLZ library, also surfaced as F1 `lib/` overrides) adding policy guardrails for sovereign/regulated workloads. |
| Sovereign Landing Zone (SLZ) offering | **I1** `Azure/sovereign-landing-zone`: the **PowerShell-orchestrated Bicep** deployment offering for Microsoft Cloud for Sovereignty. Distinct from the G1 *SLZ library* (`platform/slz` Terraform/AVM data) — I1 is a deployable Bicep accelerator that vendors ALZ-Bicep (A1) and adds confidential MGs + the Sovereignty Baseline. Being archived H1 2026. |
| Microsoft Cloud for Sovereignty (MCfS) | Microsoft's sovereign-cloud program; the SLZ (I1) is its reference landing zone. Default resource prefix `mcfs`. |
| Sovereignty Baseline policy initiatives (SBPI) | The SLZ-specific policy set definitions (`slz-CustomPolicySetDefinitions.bicep`, auto-generated by `Invoke-SlzCustomPolicyToBicep.ps1`) enforcing data residency, CMK, and confidential computing; layered on top of the ALZ Policies. |
| Confidential Corp / Confidential Online | Two of SLZ's four landing zones — the confidential variants of Corp/Online that **only allow confidential-computing resources** (protecting data in use). |
| Confidential computing | Hardware-based trusted execution (e.g. confidential VMs/AKS) protecting **data in use**; mandated on the SLZ confidential management groups. |
| Customer-managed keys (CMK) | Encryption keys controlled by the customer (Key Vault / Managed HSM); one of SLZ's three sovereignty pillars. |
| Data residency / service location | Constraining resource deployment + data storage to approved regions (`parAllowedLocations` / `parAllowedLocationsForConfidentialComputing`); SLZ's third sovereignty pillar. |
| bootstrapScopeEscape | The SLZ `bootstrap/bootstrapScopeEscape.bicep` (`targetScope='managementGroup'`) used to deploy SLZ under an **existing child management group** without tenant-root access (selected when `parTopLevelManagementGroupParentId` is set). |
| alzlibtool | A **Go** CLI in the `alzlib` repo (`go install github.com/Azure/alzlib/cmd/alzlibtool@latest`) exposing alzlib features — cache creation, library checks, and document generation (it regenerates G1's auto-generated `README.md` indexes). |
| alzlib | The `Azure/alzlib` Go library (G2): the engine that reads the ALZ Library (G1), validates it, fetches built-in policy defs from Azure, and builds the deploy-ready model consumed by the `alz` provider (G3). |
| Library model (alzlib) | alzlib's first layer — the `AlzLib` struct: name-keyed in-memory catalogs of archetypes, architectures, (versioned) policy/set definitions, assignments, role definitions, and default policy values, built by `AlzLib.Init`. |
| Deployment model (alzlib) | alzlib's second layer — the `deployment.Hierarchy` / `HierarchyManagementGroup`: a scope-resolved MG hierarchy where every asset's resource ID is rewritten to its management-group scope, built by `Hierarchy.FromArchitecture`. |
| `LibraryReference` | An alzlib interface (`FS()`, `Fetch()`, `String()`) representing one ALZ Library member; fetched via `hashicorp/go-getter` into the `.alzlib` cache, with inter-library dependencies resolved recursively. |
| Built-in policy fetch | alzlib's on-demand retrieval of **built-in** Azure policy/set definitions (via the `armpolicy` SDK) that assignments reference but the library doesn't ship; checks a `BuiltInCache` first, bounded by `Parallelism`. |
| PolicyRoleAssignment | An alzlib `deployment` struct `{ role_definition_id, scope, assignment_name, management_group_id }` describing an extra role a `Modify`/`DINE` policy identity needs; computed separately because system-assigned principal IDs aren't known until after deployment. |
| assignPermissions (metadata) | A policy-parameter metadata flag (`assignPermissions: true`) telling alzlib to also create a role assignment scoped to the resource named by that parameter's value — enabling least-privilege DINE/Modify remediation. |
| goarmfunctions | A Go library (`matt-FFFFFF/goarmfunctions`) alzlib uses to evaluate ARM template expressions embedded in policy-set parameter references when computing role-assignment scopes. |
| Policy versioning (alzlib) | First-class version support in alzlib: policy/set definitions are stored as version collections (semver), and assignments may pin a `definitionVersion`; names use the `name@version` convention (`SplitNameAndVersion`/`JoinNameAndVersion`). |
| UniqueRoleDefinitions | An `AlzLib` option (default true) that renames each role definition deterministically to `uuidV5(mgID, name)` and suffixes its display name with ` (<mgID>)` to avoid tenant-wide name collisions. |
| ALZLIB_DIR / ALZLIB_LIBRARY_GIT_URL | alzlib environment variables: the local library cache directory (default `.alzlib`) and the Git source of the ALZ Library (default `github.com/Azure/Azure-Landing-Zones-Library`). |
| terraform-provider-alz | The `azure/alz` Terraform provider (G3): a **data-source-only** provider (built on the Terraform Plugin Framework) that runs `alzlib` inside Terraform and exposes the resolved ALZ model. Designed to be consumed from within B1, not directly. |
| `alz_architecture` (data source) | The provider's core data source: given `name` + `root_management_group_id` + `location`, returns `management_groups` (flat list, per `level`, with JSON-encoded policy/role assets) and `policy_role_assignments`. Backed by `deployment.Hierarchy.FromArchitecture`. |
| `alz_metadata` (data source) | A small G3 data source returning `alz_library_references` — the list of ALZ Library refs currently loaded by the provider. |
| Data-source provider | A Terraform provider that exposes only data sources (no managed resources); the `alz` provider is one — all actual resource creation is delegated to the AzAPI provider in B1. |
| Terraform Plugin Framework | HashiCorp's modern Go SDK for building Terraform providers; `terraform-provider-alz` is built on it (with code-generated schema/model types in `internal/gen`). |
| `policy_default_values` (provider) | The `alz_architecture` map input `{ defaultName = jsonencode({ value = ... }) }` that fills G1 `defaultName` tokens across matching assignments — the Terraform surface of `Hierarchy.AddDefaultPolicyAssignmentValue` (where B2 outputs enter policy). |
| `policy_assignments_to_modify` (provider) | The `alz_architecture` nested-map input for per-MG, per-assignment overrides (parameters, enforcement_mode, identity, non_compliance_messages, overrides, resource_selectors, not_scopes) — the Terraform surface of alzlib's `ModifyPolicyAssignment`. |
| Built-in policy cache (provider) | A gzipped cache of built-in Azure policy/set definitions (`cache_file_name`/`cache_file_save_enabled`), created with `alzlibtool cache create`, that lets the provider skip Azure API calls during `Init`. |
